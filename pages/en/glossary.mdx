# Глоссарий 

#### Аккаунт

Mina использует аккаунты для отслеживания состояния каждого публичного ключа, что отличается от UTXO-модели Биткоина для поддержания состояния реестра. 

#### Архивная нода

Ноды Mina являются сжатыми по умолчанию, что означает, что им не нужно хранить историческую информацию о сети, блоке или транзакциях. [Архивная нода](/docs/advanced/archive-node) хранит данные истории сети в постоянном источнике данных, чтобы их можно было извлечь позже. 

#### Best Tip

Best tip - последний блок блокчейна с самой высокой [прочностью цепи](#chain-strength), известной daemon. 

#### Блок

Набор транзакций и согласованной информации, которые расширяют состояние сети. Включает доказательство того, что текущее состояние сети полностью соответствует действительности. 

#### Блокчейн

Структура данных, которая используется в криптовалюте для поддержания общего состояния всех аккаунтов в сети.

#### Подтверждение блоков

Количество блоков, добавленные после контрольного блока. По мере увеличения количества подтверждений уменьшается вероятность реорганизации, тем самым увеличивая вероятность подтверждения всех транзакций в контрольном блоке. 

#### Скорость заполнения блока 

Часть [слотов](#slot), которая должна содержать блок. Некоторые слоты являются пустыми преднамеренно, чтобы сеть могла "наверстать упущенное" в случае задержки сообщений. 

#### Блок-продюсер

Нода, которая участвует в процессе определения, какие блоки ей разрешено создавать, а затем создает блоки, содержащие транзакции, которые могут транслироваться в сеть. 

#### Bootstrap

Бутстрэппинг - это часть процесса [синхронизации](#syncing) ноды. При бутстрэппинге пиры предоставляют текущий [корень](#root-of-transition-frontier) [границы перехода](#transition-frontier). Дополнительные [переходы](#transition), полученные в процессе [наверстывания или catchup](#catchup), используются из этого начального корневого состояния. 

#### Breadcrumb

Breadcrumb – нода на [границе перехода](#transition-frontier). Breadcrumb содержит внешний переход, поэтапный реестр (staged ledger) и ожидающие монетные базы и создается при переходе к предыдущему состоянию. 

#### Catchup

Catchup - это заключительный этап процесса [синхронизации](#syncing). Во время catchup нода будет пытаться догнать текущий [best tip](#best-tip), определяя, а затем загружая все [переходы](#transition) между [корнем](#root) границы перехода и текущим best tip. Сначала нода запрашивает недостающие хэши перехода и доказательство цепочки транзакций. Это доказательство доказывает, что указанный путь действителен, т.е. что предоставленные хэши перехода ведут от корня к best tip. Как только нода получит все хэши перехода, она запрашивает у пиров полный внешний переход для каждого хэша перехода. 

With each external transition, the node builds up its transition frontier by applying each to the prior state to construct a [breadcrumb](#breadcrumb). When catchup is complete, the node's local best tip is the same as the network's best tip, and breadcrumbs have been constructed for all transitions from the transition frontier root (best tip - k) to the current tip, and each has been validated. At this point, the node is [synced](#synced).

At any time, a catchup may be triggered if the node sees a disjoint transition in the same path, indicating there are missing transitions.

#### Chain Strength

As there is no full history available in Mina, a newly connected node to the network cannot sync from genesis by applying all prior transitions. To allow a node to determine the strongest chain, a minimum chain density is stored for a sliding window of time. As a result, honest nodes may choose the blockchain with the higher minimum density or chain strength.

#### Compressing

Generating a SNARK for a computation output can be thought of as "compressing" that output, as the proofs are fixed size. For example, Mina maintains a succinct blockchain by compressing all the historical data in a blockchain into a zk-SNARK. However, this is computationally different from lossy compression, and the term compress is used to more figuratively describe the process of reducing the size of data required.

#### Consensus

An algorithm or set of rules that Mina nodes all agree upon when deciding to update the state of the network. Rules may include what data a new block can contain and how nodes are selected and rewarded for adding a block. Mina implements Ouroboros Samisika as its consensus mechanism.

#### Consensus Node

A participant in the Mina network that performs the consensus function, i.e., a [block producer](#block-producer). For a node to participate in consensus, it requires the recent state.

#### Cryptocurrency

A digital asset or currency that uses cryptographic primitives to secure financial transactions and to verify ownership via public-private key pairs.

#### Daemon

The Mina daemon is a background process that implements the Mina protocol and runs on a node locally. This allows a local client or wallet to talk to the Mina network. For example, when a CLI is used to issue a command to send a transaction, this request is made to the Mina daemon, which then broadcasts it to the peer-to-peer network. It also listens for events like new blocks and relays this to the client via a [pub-sub](#pub-sub) model.

#### Delegating

Because staking mina requires nodes to be online, some nodes may desire to delegate their mina to another node that runs a staking service. This process is called delegating a stake, and often the service provider or staking pool operator may charge a fee for running this service, which will be deducted any time the delegator gets selected to be a block producer.

#### Epoch

An epoch is a unit of time equal to 7140 slots at Mainnet. An epoch is divided into [slots](#slot) of 3minutes each.

#### External Port

This is the port that the Mina daemon uses to connect to other nodes on the network. It can be set using `-external-port` when starting the daemon.

#### External Transition

An external transition also referred to as a [block](#block), is a transition that is generated externally, i.e., by another block producer, and gossiped to a node.

#### Finality

A consensus constant `k` the point at which chain [reorganizations](#reorganizations) are no longer possible. Once a block has `k` [block confirmations](#block-confirmations), defined by the consensus constants, it is considered final.

#### Full Node

A Mina node that is able to verify the state of the network trustlessly. In Mina, every node is a full node since all nodes can receive and verify zk-SNARKs.

#### Internal Transition

A [transition](#transition) that is produced locally, i.e., by a block producer. The generated transition is applied locally and added to the [transition frontier](#transition-frontier) before being broadcast to peers.

#### Kademlia

A distributed hash table (DHT) for decentralized peer-to-peer networks. Mina uses Kademlia for peer discovery so that nodes can find neighbor nodes to share information about the network state.

#### Keypair

A combination of a [private](#private-key) and [public](#public-key) key. Keypairs may be generated via a running daemon or using a dedicated keygen tool. 

#### Mina

- "Mina" with a capital M references the underlying cryptocurrency protocol and the network infrastructure that the system depends upon
- "mina" is the unit of the cryptocurrency that is exchanged by participating nodes on the network

#### Node

A node is a machine running the mina daemon.

#### Non-consensus Node

A [full node](#full-node) in the Mina protocol that does not participate in consensus but can still fully verify the zero-knowledge proof to trustlessly validate the state of the chain. When referring to the size of Mina as 22kb, this is in reference to non-consensus nodes.

#### Nonce

An incrementing number attached to a transaction used to prevent a replay of a transaction on the network. Transactions are always included in blocks in the sequential order of the nonce.

#### Peer-to-peer

Networking systems that rely on peer nodes to distribute information amongst each other are called peer-to-peer networks. These networks are often distributed in nature, and unlike client-server networking models, do not rely on any centralized resource broker.

#### Pickles SNARK

Pickles is a new proof system and associated toolkit that is the first deployed SNARK capable of recursive composition with no trusted setup. Pickles serves as the basis for developers to build private, scalable smart contracts on Mina. [Read more](https://medium.com/minaprotocol/meet-pickles-snark-enabling-smart-contract-on-coda-protocol-7ede3b54c250).

#### Libp2p

Сетевая p2p-библиотека, обеспечивающая такие вещи, как транслирование сообщений и обмен файлами. Mina использует ее для ознакомления пользователей. 

#### Приватный ключ

Другой компонент криптографии с использованием публичного ключа - приватный ключ (закрытый ключ) – хранится в секрете, в то время как публичные ключи могут быть предоставлены кому-либо. Только владелец публичного ключа, соответствующего приватному ключу, может подтвердить право собственности на публичный ключ. Это позволяет подписывать транзакции с целью доказать, что вы являетесь честным держателем любых средств, связанных с любым таким публичным ключом. 

#### Состояние протокола

Состояние сети. Состояние протокола включает в себя хэш предыдущего состояния протокола для связывания блоков вместе, а также тело, содержащее хэш состояния генезиса, состояние цепочки блоков, состояние консенсуса и константы консенсуса. 

#### Хеш состояния протокола 

Хэш состояния протокола - это хеш хэшей предыдущего состояния и тела [состояния протокола](#protocol-state). Действует в качестве уникального идентификатора блока. 

#### Proof-of-Stake

Тип алгоритма консенсуса, используемый Mina, позволяет нодам согласовывать состояние сети. Proof-of-Stake (PoS) позволяет нодам осуществлять [стейкинг](#staking) mina в сети, увеличивая свои шансы быть выбранным в качестве следующего блок-продюсера. 

#### Публичный ключ

Один из компонентов криптографии с использованием публичного ключа - публичный ключ (открытый ключ) – может широко использоваться в мире и считаться "адресом" или идентификатором человека, который владеет соответствующим приватным ключом. 

#### Pub-sub

Pub-sub - это сокращение от слова "публикация-подписка". Это шаблон обмена сообщениями, в котором отправители сообщений транслируют сообщения, и все слушатели, которые ранее подписались на сообщения этого отправителя, будут о них уведомлены. Mina, например, использует этот шаблон в качестве способа уведомления клиентов о добавлении нового блока в блокчейн. Об этом событии могут "услышать" все слушатели, и каждому независимому слушателю не нужно запрашивать новые данные. 

#### Реорганизация 

Когда конкурирующий форк блокчейна увеличивается по длине относительно основной ветки, блокчейн подвергается реорганизации, чтобы отразить более сильный форк в качестве главной ветки. После реорганизации не гарантируется включение транзакций на отвергнутой ветке в блокчейн, и их нужно будет добавлять в новые блоки на самой длинной ветке. 

#### Корень (границы перехода)

Корень [границы перехода](#transition-frontier) - это блок `k` блоков из [best tip](#best-tip). Корень может быть получен пирами во время [бустрэпа](#bootstrap). Как только появляется новый best tip, корень перемещается, поэтому остаются только блоки k. Корень - это точка, в которой блок был [завершен](#finality) в результате консенсуса. 

#### RPC

Удаленный вызов процедур [(RPC)](https://en.wikipedia.org/wiki/Remote_procedure_call) используется для взимодействия между нодами в сети, а также для взаимодействия с запущенным [daemon](#daemon).

#### Scan State

[Scan state](/docs/architecture/scan-state) – это структура данных, которая ставит в очередь транзакции, требующие подтверждения SNARK-доказательств транзакций, и позволяет выполнять параллельную обработку этих SNARKS транзакций [снарк-воркерами](#snark-worker). 

#### Подписи

Сокращение цифровых подписей. Подписи – это способ установить подлинность или право собственности на сообщения с цифровой подписью. Это возможно, потому что при наличии keypair, состоящего из публичного и приватного ключей, владелец приватного ключа может подписать сообщение и создать подпись, которую может проверить каждый, у кого есть связанный публичный ключ. 

#### Слот

Слот в Mina – это единица времени. На момент запуска Mainnet продолжительность слота составляет 3 минуты. [Эпоха](#epoch) делится на слоты. Блок-продюсеры могут выиграть возможность произвести блок во время слота. 

#### SNARK Pool

Снарк-пул содержит работу, выполненную [снарк-воркерами](#snark-worker) для необходимой работы в структуре данных [scan state](#scan-state). Снарк-пул, также называемый "снаркетплейсом", содержит только самые дешевые работы, предлагаемые снарк-воркерами для каждого комплекта работ. Если нода видит более дешевую и валидную работу, она заменит ею уже существующую в пуле. 

#### SNARK Worker

Нода в сети, участвующая в генерации SNARK для транзакций. Снарк-воркер заинтересован в том, чтобы оказывать помощь в сжатии транзакции, потому что он получает mina в качестве вознаграждения за свои усилия. 

#### Snarked ledger

Snarked ledger – это реестр, который содержит только транзакции, имеющие связанное доказательство. Snarked ledger обновляется после того, как из [scan state](#scan-state) было отправлено доказательство, подтверждающее все транзакции, включенные в дерево транзакций, добавленных благодаря предыдущим [блок-продюсерам](#block-producers). 

#### Staged Ledger

Поэтапный реестр (staged ledger) – это текущее состояние аккаунта. Он содержит реестр ожидающих аккаунтов, а также очередь из ожидающих необработанных SNARK-транзакций, известную как [scan state](#scan-state). 

#### Стейкинг

Стейкинг mina позволяет нодам в сети увеличивать свои шансы быть избранными в качестве блок-продюсеров в соответствии с механизмом консенсуса. Шанс выигрыша блока увеличивается пропорционально размеру стека mina. Например, если одна нода отправляет в стейкинг 50% доступных mina в сети, у нее теоретически есть 50% шанс быть избранной для создания следующих блоков. Mina использует Proof-of-Stake Ouroboros Samisika для реализации деталей стейкинга. Если нода решает отпавить свои mina в стейкинг, она должна находится онлайн и быть подключенной к сети Mina. 

#### Реестр стейкинга

Реестр стейкинга – это реестр, который используется при выборе блок-продюсеров для слота, поскольку вероятность выигрыша слота пропорциональна сумме стека. Распределение стека текущей эпохи – это [snarked-ledger](#snarked-ledger) последнего блока текущей эпохи - 2. 

#### Стейкинг-пул

Пул делегированных средств, которыми управляет владелец стейкинг-пула. Другие ноды могут делегировать средства в стейкинг-пул, чтобы избежать необходимости быть онлайн. 

#### Синхронизация

Чтобы успешно создать блок, который расширяет блокчейн Mina, ноде необходимо текущее состояние. Для этого нода инициализируется для подключения к пирам, использует [бустрэп](#bootstrap), а затем выполняет [наверстывание](#catchup). 

В процессе синхронизации создается [граница перехода](#transition-frontier) ноды путем создания [хлебных крошек](#breadcrumb) для всех переходов между [корнем](#root-of-transition-frontier) границы перехода и текущим [best tip](#best-tip). По завершении узел синхронизируется. 

#### Временно заблокированные аккаунты

Аккаунт с необеспеченным количеством токенов, которые нельзя перемещать до тех пор, пока не будет выполнено определенное условие, такое как производство определенного количества блоков. 

#### Пул транзакций 

Список всех ожидающих транзакций – иногда называется "мемпулом". У каждой ноды есть локальный пул транзакций, который содержит транзакции, которые были переданы ноде и подтверждены. Если пул транзакций заполняется, транзакции, находящиеся в пуле, могут быть исключены в соответствии с комиссией – будет исключена самая низкая сумма. 

#### Переход

Переход в Mina является синонимом [блока](#block). 

#### Граница перехода

Локальное хранилище данных, которое содержит последние `k` блоки сети. Это структура данных многостороннего дерева, поскольку каждая нода дерева может иметь несколько дочерних элементов, то есть форков. Каждая нода в дереве называется [хлебной крошкой или breadcrumb](#breadcrumb). 

#### Пользовательская транзакция 

Транзакция, совершенная пользователем – либо платеж, либо изменение делегирования 

#### VRF

Проверяемая случайная функция (VRF) используется при выборе блок-продюсера для слота. В качестве входных данных она принимает случайный сид, полученный из выходных данных VRF предыдущей эпохи, в дополнение к публичному ключу и текущему реестру стейкинга, и является детерминированной, то есть она всегда будет возвращать один и тот же результат независимо от того, как часто она запускается. 

#### Доказательство с нулевым разглашением 

Доказательство, с помощью которого одна сторона (доказывающий) может доказать другой стороне (проверяющим), что они что-то знают, при этом не передавая эти конкретные знания. Mina использует доказательства с нулевым разглашением, в частности, zk-SNARK, для создания доказательства, подтверждающего валидность блокчейна и позволяющего любой ноде в сети быстро это проверить. 

#### zk-SNARK

Тип доказательства с нулевым разглашением – акроним, представляющий собой "сжатый неинтерактивный аргумент знания с нулевым разглашением". Особые свойства, вызывающие интерес к SNARK, реализованных в Mina, – это сжатость и отсутствие интерактивности, что позволяет любой ноде быстро проверять состояние сети.
